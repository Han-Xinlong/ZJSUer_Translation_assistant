# ZJSUer Translation Assistant

面向外语学习者的“随写随翻”AI 工具开发与实践应用。

本项目源自浙江工商大学东方语言与哲学学院 2026 年度大学生创新创业训练计划项目，目标是打造一款轻量化、可复盘、会督促的 AI 写作与翻译学习伙伴，帮助外语学习者在日常写作、翻译和修改过程中真正积累表达能力，而不是只得到一次性的机器翻译结果。

## 项目背景

外语学习者在写作和翻译训练中长期面临三类问题：

- **写不出来**：词汇储备不足，句式单一，表达启动困难。
- **翻不地道**：普通翻译工具容易机械直译，缺少语境感知和表达优化。
- **改完记不住**：AI 给出修改结果后，学习者往往缺少复盘机制，难以内化为自己的语言能力。

现有工具也存在明显空缺：专业 CAT 平台功能强但学习成本高，简单词典或轻量翻译应用又无法满足写作、润色、复盘、积累等多维需求。因此，本项目希望把 AI 从“一次性工具”升级为“长期学习伙伴”。

## 产品定位

**轻量陪伴，深度成长。**

ZJSUer Translation Assistant 不是一个复杂的专业翻译平台，而是聚焦个人日常外语学习场景的 AI 辅助工具。它强调：

- 轻量化：打开即写，随写随查，降低使用门槛。
- 可复盘：记录“写作 - 翻译 - 修改 - 积累”的完整过程。
- 会督促：通过目标、提醒、打卡和学习数据，帮助用户形成持续练习习惯。

## 核心功能规划

### 1. 智能输入

- 支持文本输入与语音录入。
- 提供沉浸式写作界面。
- 在写作过程中保持 AI 随时待命，方便用户快速求助。

### 2. 随写随翻

- **快速模式**：适合秒查词句、快速获得译文。
- **深度模式**：采用“初译 + 审校”的双模型/双阶段协作思路，提升译文准确性、自然度与语境贴合度。
- 根据用户输入智能推荐相关语料与优秀表达。

### 3. 随写随修

- 提供基础语法纠错。
- 支持风格、句式、逻辑和表达自然度优化。
- 展示修改说明，帮助用户理解“为什么这样改”。

### 4. 成长可视化

- 学习热力图。
- 历史写作记录。
- 初稿 vs 终稿对比。
- 自动沉淀错题库、常见错误库和优秀表达库。

### 5. 社群互学

- AI 实时答疑。
- 优质语料与表达共享。
- 好友进度对比与学习激励。

## 核心创新

### 双模式翻译的轻量化应用

将“快速/深度”两个入口前置给用户，让学习者根据任务场景决定 AI 介入深度。快速模式解决即时查询，深度模式服务高质量翻译与写作训练。

### 学习过程数据化与可复盘

工具不只输出结果，还记录用户从初稿到译文、从修改建议到最终版本的全过程。通过错误沉淀和优秀表达积累，减少对 AI 的被动依赖，推动知识内化。

### 游戏化学习激励

通过目标设定、每日提醒、打卡成就、好友进度对比等机制，把枯燥练习转化为可持续的习惯养成。

### “教师 - 学生”成果创新链

项目源自指导教师的教学研究，由学生团队开发成应用，再反哺写作与翻译课程教学，形成“教学研究 -> 工程实践 -> 学习应用”的闭环。

## 技术路线

项目计划采用“前端轻量、后端服务化、数据本地化”的架构。

### 前端

- 单页 Web 应用：Vue 或 React。
- 富文本编辑器：支持写作、翻译、批注和版本对比。
- 可视化能力：使用 ECharts 展示学习热力图、练习趋势和错误分布。

### 后端

- Python FastAPI。
- 封装 AI API 调用。
- 编排快速模式、深度模式、润色模式等不同 AI 工作流。

### AI 模型协作

- 支持接入通用大模型 API，如 GPT、文心、通义等。
- 深度模式采用“初译 + 审校”的协作流程。
- 后续可根据语言方向和任务类型优化 Prompt 模板。

### 数据存储

- 本地存储优先：IndexedDB 或 SQLite。
- 云端仅用于可选备份，尽量保护用户隐私。
- 重点保存写作历史、修改记录、错题库、表达库和学习进度。

## 本地开发

建议环境：

- Python 3.10 优先，Python 3.8 也可运行当前后端骨架。
- Node.js 16+。
- npm 首次安装依赖时可能需要较长时间下载 Vite、ECharts、Babel 等依赖，请耐心等待输出。

### 给非开发同学的快速体验方式

如果只是想从 GitHub 下载项目并体验前端交互，推荐直接阅读这份说明：

- [给同学体验前端交互的快速指南](docs/classmate_quick_start.md)

最短路径是打开两个终端，分别运行：

```bash
bash scripts/dev_backend.sh
```

```bash
bash scripts/dev_frontend.sh
```

然后在浏览器访问：

```text
http://localhost:5173
```

后端默认是 `mock` 模式，不需要配置真实 AI Key，也可以体验翻译、润色、收藏表达、错题库、历史详情和学习档案等完整流程。

### 启动后端

```bash
bash scripts/dev_backend.sh
```

后端默认运行在 `http://localhost:8000`。

常用检查接口：

- `GET /api/health`：基础健康检查。
- `GET /api/status`：查看当前 AI Provider、模型和配置状态。

脚本会优先使用 `python3.10` 创建虚拟环境；如果本机暂未安装 Python 3.10，则回退到 `python3`。

### 启动前端

```bash
bash scripts/dev_frontend.sh
```

前端默认运行在 `http://localhost:5173`，开发环境会将 `/api` 请求代理到后端服务。

> 当前仓库处于从 0 到 1 的初始化阶段，已完成基础目录、前端工作台雏形、FastAPI 服务骨架和 Prompt 模板占位。后续将继续补充真实模型调用、富文本编辑、本地学习数据存储和可视化能力。

### 验证项目

```bash
npm --prefix frontend test
npm --prefix frontend run build
cd backend && .venv/bin/python -m pytest
python3 -m compileall backend/app
```

### 启用真实 AI 调用

后端默认使用 `mock` 模式，便于无密钥演示。若要接入 OpenAI：

```bash
cp backend/.env.example backend/.env
```

然后在 `backend/.env` 中配置：

```env
AI_PROVIDER=openai
OPENAI_API_KEY=your_api_key
OPENAI_MODEL=gpt-5-mini
```

配置完成后重新启动后端即可。当前后端使用 OpenAI Responses API，并通过 Prompt 要求模型返回 JSON，方便前端稳定展示翻译、审校和润色结果。前端右侧会展示当前是 `mock` 演示模式还是真实模型模式。
